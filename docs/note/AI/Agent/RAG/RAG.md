## 1. 什么是 RAG

RAG，Retrieval-Augmented Generation，中文通常叫“**检索增强生成**”。它的核心思想很简单：

**先查资料，再回答问题。**

传统大模型主要依赖参数里“记住”的知识来回答；RAG 会先从外部知识库中检索相关内容，再把检索到的内容作为上下文交给大模型生成答案。这样做的价值在于：

- 知识可以更新，不必每次都重新训练模型
- 能接入企业私有数据、专业文档、实时资料
- 能明显降低无依据胡编的概率
- 回答更容易做到“有出处、有证据”

一句话概括：
**RAG = 检索系统 + 大模型生成系统。**

## 2. 为什么需要 RAG
纯 LLM 回答问题时，常见有四类问题：

### 2.1 知识过时
模型训练数据有截止时间，之后发生的新事实，它未必知道。RAG 可以把外部最新资料接进来。

### 2.2 私有知识缺失
企业内部文档、制度、项目资料、产品手册，通常不在模型训练集中。RAG 正好解决“模型不知道你自己的资料”这个问题。

### 2.3 幻觉
模型有时会生成看起来合理、实际没有依据的内容。RAG 通过先提供证据，能显著提升回答的可验证性。原始 RAG 论文和后续评测都把这一点作为重要优势。

### 2.4 领域问答精度不够
医学、法律、金融、工业文档这类场景，对术语和细节要求很高。单靠参数记忆不稳，必须先查领域资料再答。

## 3. RAG 的本质结构
从系统角度看，RAG 其实就是三层：

### 3.1 知识准备层
负责把原始文档变成“可检索”的知识单元。

### 3.2 检索层
负责从知识库里找到和用户问题最相关的证据。

### 3.3 生成层
负责基于证据组织答案，而不是脱离证据自由发挥。

所以 RAG 不是一个单点技术，而是一条完整流水线。


## 4. RAG 的全流程

标准 RAG 可以拆成下面 8 步。

### 4.1 数据接入
把原始资料接进系统，来源可能包括：

- PDF
- Word
- Markdown
- HTML / 网页
- 数据库记录
- FAQ
- API 返回内容
- 表格
- 代码仓库
- 企业知识库

这一层最重要的是：**先把数据清洗干净。**

因为如果原始文档里有大量噪声，比如页眉页脚、目录、重复导航、表格错位、乱码，后面检索效果会大幅下降。

### 4.2 文档解析与清洗
把文档内容转成结构化文本或半结构化内容。

常见处理包括：
- 去除页码、页眉、页脚
- 去掉重复标题
- 保留标题层级
- 保留列表结构
- 保留表格边界
- 保留代码块边界
- 识别章节、段落、问答对

核心原则：

**不是把文档“变成一坨纯文本”，而是尽量保留它原有的知识结构。**

### 4.3 文本分块（Chunking）
这是 RAG 中最关键的环节之一。

因为大模型和 embedding 模型通常不是直接处理整篇文档，而是处理文档块。  
分块的目标是：

**把长文档切成适合检索、又尽量语义完整的小单元。**

#### 常见分块策略

#### 1.固定长度分块
按字数或 token 数切，比如每 500 token 一块。

优点：
- 简单
- 稳定
- 易实现

缺点：
- 容易切断句子
- 容易切断定义与解释
- 块边界不自然

#### 2.带 overlap 的固定分块
相邻块保留部分重复内容。

例如：
- 块 1：1–500
- 块 2：450–950

优点：
- 缓解边界切断问题
- 保住部分上下文连续性

缺点：
- 数据重复
- 存储和检索成本上升

#### 3.按句子/段落分块
优先按自然语言边界切，而不是按长度硬切。

优点：
- 语义更自然
- 更适合问答型文本

缺点：
- 长度不均匀
- 长段落仍要二次拆分

#### 4.按文档结构分块
按标题、小节、FAQ、函数、表格、条款等结构单元切。

这是很多生产系统里效果最好的策略之一。

#### 5.语义分块
先分句，再根据句子间语义相似度决定是否放到同一个 chunk 里。

它的核心思想是：

**边界应该服从语义，而不是只服从长度。**

#### 6.多尺度分块
同时建立小块、中块、大块索引，查询时按问题类型动态选取。

这适合复杂系统，因为：

- 事实型短问适合小块
- 分析型问题适合大块
- 背景型问题适合中大块

#### 分块的核心原则
直接记这三句：
- 一块内容尽量表达一个相对完整的知识单元
- 不能太碎，否则上下文不够
- 不能太大，否则主题太杂、噪声太多

### 4.4 向量化（Embedding）
分块后，要把每个 chunk 变成向量。

embedding 模型的作用是：

**把文本映射为数学向量，让语义相近的内容在向量空间里更靠近。**

例如：
- “什么是深度学习”
- “深度学习的定义是什么”

虽然字面不同，但语义接近，所以向量也应接近。

#### embedding 模型和 LLM 的区别
embedding 模型：
- 负责表示文本
- 负责相似度计算
- 负责检索

LLM：
- 负责理解上下文
- 负责推理
- 负责生成答案

所以：**embedding 负责“找”，LLM 负责“说”。**

### 4.5 建立索引
把 chunk 向量存进向量数据库或检索系统中。

常见索引内容包括：
- chunk 文本
- embedding 向量
- 文档来源
- chunk 位置
- 标题/章节
- 元数据（作者、时间、标签、权限等）

这里除了向量索引，还常常会保留：
- 关键词倒排索引
- BM25 索引
- 元数据过滤索引

因为纯向量检索并不总是最稳。

### 4.6 查询理解与查询改写
用户提问进入系统后，不一定直接检索。

很多时候原始 query 有问题：
- 太短
- 太模糊
- 太复杂
- 用词和文档表达不一致

所以可以先做查询转换。
#### 常见方法
#### 1.查询重写
把模糊问题改写得更具体。
例如：
- 原问题：机器学习算法
- 改写后：常见机器学习算法的分类、原理及应用场景

#### 2.步退查询
当问题太具体时，先退一步查背景。

例如：
- 原问题：如何训练 BERT
- 步退后：NLP 模型训练的基础方法与 Transformer 训练流程

#### 3.子查询分解
复杂问题拆成多个小问题分别检索。

例如：
- AI 对就业的影响  
    可拆成：
- 哪些岗位被替代
- 哪些岗位被创造
- 哪些行业受影响最大
- 如何应对就业变化

查询转换的本质是：

**优化用户问题，让它更适合检索系统理解。**

### 4.7 检索（Retrieval）
这是 RAG 的“找资料”阶段。

最基础做法是：
- 把 query 做 embedding
- 和所有 chunk 向量算相似度
- 返回 top-k

#### 为什么常用余弦相似度
因为向量检索更关心方向是否接近，而不是数值绝对大小是否一致。

所以：**余弦相似度衡量的是语义方向的接近程度。**

#### 检索的常见形式
#### 1.Dense Retrieval
纯向量检索。  
优点是能理解语义。  
缺点是对专有名词、缩写、数字编号不一定稳定。

#### 2.Sparse Retrieval
如 BM25、关键词检索。  
优点是对术语、实体名、关键词非常敏感。  
缺点是语义泛化能力差。

#### 3.Hybrid Retrieval
把 dense 和 sparse 结合。

这通常是更稳的生产方案，因为：
- dense 负责“懂意思”
- sparse 负责“抓硬信号”

很多现代检索系统都在用这种思路。

### 4.8 重排序（Rerank）
第一轮检索拿到的 top-k，不一定就是最适合回答问题的 top-k。

所以很多系统会做第二轮精排。

#### 为什么要 rerank
因为第一轮检索更偏“召回”，目标是不要漏；  
而 rerank 更偏“精度”，目标是排得更准。

#### 常见 rerank 方式
#### 1.关键词重排
根据 query 中的关键词、实体、概念在文档中的匹配程度重新排序。

优点：
- 快
- 便宜

缺点：
- 语义理解弱

#### 2.Cross-encoder / 深度重排
把 query 和候选文档一起输入重排模型，让模型直接判断相关性。

优点：
- 准确率高

缺点：
- 慢
- 成本更高

#### 3）LLM 重排
让大模型判断“这段内容和问题有多相关”。
优点：
- 判断能力强

缺点：
- 最贵
- 延迟高

#### 4.混合重排
综合：
- 原始相似度
- 关键词分数
- rerank 分数
- 元数据质量信号

这是比较实用的工程方案。

一句话记忆：

**检索负责找得到，rerank 负责排得对。**

### 4.9 上下文构建（Context Building）

拿到 top-k 后，不是直接拼接完事。

因为实际问题常常不只依赖某一个 chunk，而依赖：

- 相邻上下文
- 同一章节的解释
- 定义 + 示例
- 条款 + 例外情况

所以要做上下文优化。

#### 常见方式

#### 1.邻居扩展
命中一个 chunk 后，把它前后相邻块也带上。

适合：
- 论文
- 手册
- 技术文档
- 流程说明

#### 2.标题注入
把 chunk 所属标题、章节名、文档名一起加入上下文。

这样模型更容易知道这段内容“属于什么主题”。

#### 3.问题增强文档
提前为每个 chunk 生成可能问题，检索命中这些问题后，再回到原始 chunk 作为证据。

这种方式本质上是在增强“可检索表达”。

#### 4.段落重构
不直接使用离散 chunk，而是把高价值相邻块重新拼成更完整的段落。
这能减少碎片化问题。

一句话概括：**最终给模型的，不应该只是几段“像是相关”的碎片，而应该是足够完整、足够连贯的证据上下文。**

### 4.10 Prompt 构建
拿到上下文后，要设计提示词。

一个合格的 RAG prompt 通常要包含三点：

#### 1.限制信息来源
明确要求模型只能基于给定上下文回答。

#### 2.允许拒答
如果上下文不足，就明确说不知道，而不是猜。

#### 3.鼓励引用
如果产品需要可追溯性，可以要求模型引用来源。

例如可以这样设计：
- 你是一名基于资料回答问题的助手
- 只能根据提供的上下文回答
- 若上下文无法支持答案，明确说明信息不足
- 回答时优先简洁、准确
- 必要时给出引用片段或来源编号

这一步非常重要，因为很多幻觉不是检索没找到，而是 prompt 没有给模型“边界”。

### 4.11 答案生成
最终由 LLM 基于：
- 用户问题
- 检索上下文
- 生成提示词

来输出答案。

这里生成阶段的目标不是“最会说”，而是：
- 忠于证据
- 结构清晰
- 信息完整
- 不乱推断

所以高质量 RAG 的核心不是文风，而是**有根据地回答**。

## 5. RAG 的进阶优化方案
如果要把 RAG 做得更强，通常沿着下面几条路线优化。

### 5.1 分块优化
这是最值得优先投入的地方。

优化方向包括：
- 固定分块 → 语义分块
- 单尺度分块 → 多尺度分块
- 只存正文 → 存正文 + 标题 + 元数据
- 无脑 overlap → 按文档类型自适应 overlap

很多系统提升效果，不是靠换更大模型，而是靠改 chunk。

### 5.2 查询优化
适合用户问题变化大、问法不稳定的场景。

常见策略：
- query rewrite
- step-back query
- sub-query decomposition
- 同义词扩展
- 实体标准化
- 意图识别

本质上是在解决“用户不会刚好用知识库里的表达方式提问”这个问题。

### 5.3 检索优化
可以从这几方面做：
- dense + sparse 混合检索
- 多路召回
- metadata filter
- 按权限过滤
- 按时间过滤
- 文档级先召回，再 chunk 级精排

对于企业文档库，这一层非常重要，因为很多问题不是“相关性”，而是“权限和范围”。

### 5.4 重排优化
可以逐步升级：
- 无重排
- 规则重排
- cross-encoder rerank
- LLM rerank
- 混合多信号 rerank

如果预算允许，rerank 往往是提升精度最直接的方式之一。

### 5.5 上下文优化
可做：
- 邻居扩展
- 连续段落重构
- 章节聚合
- 证据去重
- 长度控制
- 多文档融合
- 冲突证据处理

这一层经常决定最终回答是否“像人写的、像读过资料后写的”。

### 5.6 生成优化
包括：
- 更严格的系统提示词
- 结构化回答模板
- 强制引用
- 先摘要证据再回答
- 先列证据再推理
- 多轮问答的会话记忆管理

## 6. 高质量 RAG 的典型架构

一个成熟的 RAG 系统，通常不是“一个向量库 + 一个模型”这么简单，而更像下面这样：

### 离线阶段
- 文档接入
- 清洗解析
- 结构提取
- 分块
- 生成 embedding
- 建立向量索引
- 建立关键词索引
- 生成 chunk 头部/摘要/问题增强
- 保存元数据

### 在线阶段
- 接收 query
- 查询理解与改写
- 多路召回
- 合并候选
- rerank 精排
- 上下文构建
- prompt 构建
- LLM 生成
- 返回答案与来源

### 评估阶段
- 检索质量评估
- 生成质量评估
- 用户反馈回流
- 失败案例分析
- 持续优化索引与提示词


## 7. RAG 的评估体系
很多人做 RAG 最大的问题，是只看“回答像不像对”，却不拆开看系统到底哪一层出了问题。

RAG 至少要分两层评估：

### 7.1 检索评估
看的是“资料找得对不对”。
常见指标：
- Recall@K
- Precision@K
- MRR
- NDCG
- Top-k 是否命中黄金证据

### 7.2 生成评估
看的是“回答说得对不对”。

常见维度：
- Accuracy
- Relevance
- Faithfulness
- Completeness
- Citation correctness

RAG 的关键不是只评最终答案，而是：

**检索和生成要分开评。**

因为如果回答错了，可能是：
- 没检到
- 检到了但排序错了
- 排对了但上下文构建差
- 上下文对了但模型乱说

## 8. RAG 常见失败原因
### 8.1 数据质量差
源文档解析错了，后面全错。

### 8.2 分块错误
块太大、太小、边界乱切，都会影响召回。

### 8.3 只做纯向量检索
忽略关键词、实体、缩写、编号，导致漏召回。

### 8.4 没有 rerank
top-k 结果表面相似，实际不最相关。

### 8.5 上下文太碎
给模型的是碎片，不是证据链。

### 8.6 prompt 没设边界
模型开始脑补。

### 8.7 缺少评估
系统优化靠感觉，而不是靠指标。

## 9. RAG 与长上下文的关系
长上下文模型变强后，很多人问：是不是不需要 RAG 了？

答案通常是：**不是替代关系，而是互补关系。**

长上下文解决的是：
- 你已经知道要给模型看哪些内容

RAG 解决的是：
- 面对大规模资料，系统先帮你找到该看的内容

所以在大知识库、私有知识、动态更新知识场景下，RAG 仍然非常重要。

## 10. RAG 适合哪些场景
最适合：
- 企业知识库问答
- 客服问答
- 文档助手
- 内部制度查询
- 法规/合同/手册问答
- 技术文档问答
- 医疗/金融等领域知识辅助
- 可追溯问答系统

不一定适合：
- 纯开放式创作
- 完全不依赖外部知识的任务
- 知识规模极小、直接写进 prompt 就够的场景
- 数据质量极差、解析成本极高但收益低的场景
