# Node.js

Node.js 不是一门新语言，也不是前端框架，而是一个让 JavaScript 脱离浏览器、直接运行在操作系统上的运行时（Runtime）。它的核心价值在于：**用单线程 JavaScript 配合底层的非阻塞 I/O，处理大量并发请求**。

---

## 1. Node.js 到底解决了什么问题

传统服务端模型里，常见做法是：
- 一个请求对应一个线程；
- 线程在等待数据库、文件、网络返回时会阻塞；
- 并发一高，线程切换和内存开销都会很重。

Node.js 走的是另一条路线：
- **JavaScript 主线程只有一个**；
- **I/O 不阻塞主线程**；
- 遇到文件、网络、数据库等耗时操作时，把工作交给底层系统和 `libuv`；
- 等结果准备好后，再把回调放回事件循环中执行。

所以 Node.js 的优势不在于“计算特别快”，而在于：

> **主线程不傻等 I/O，而是不断处理新的请求和回调。**

---

## 2. 单线程为什么还能扛并发
这里的“单线程”，说的是：
- 你的 JavaScript 业务代码主要在一个主线程里执行；
- 同一时刻只会有一段 JS 在主线程上跑；
- 因此没有多线程共享内存带来的锁竞争问题。

但 Node.js 又不是“只有一个线程”：
- V8 负责执行 JavaScript；
- `libuv` 负责事件循环、I/O 调度、线程池；
- 某些文件 I/O、DNS、`crypto`、`zlib` 等任务会走线程池；
- 网络 I/O 更多依赖操作系统的异步能力。

所以更准确的说法是：

> **Node.js 的 JavaScript 执行模型是单线程的，但底层并不是只有一个线程。**

---

## 3. Node.js 的核心：事件循环

Node.js 之所以能做到“主线程不阻塞”，关键在于 **事件循环（Event Loop）**。

你可以把它理解成一个持续运转的调度器：
1. 先执行当前同步代码；
2. 遇到异步任务就先登记；
3. 等异步任务完成后，把对应回调安排到合适的队列；
4. 主线程按固定规则不断取出回调执行。

Node.js 的事件循环基于 `libuv`，一轮循环里有 6 个主要阶段：

1. `timers`
2. `pending callbacks`
3. `idle, prepare`
4. `poll`
5. `check`
6. `close callbacks`

其中：
- `setTimeout` / `setInterval` 主要在 `timers`；
- I/O 回调大多在 `poll`；
- `setImmediate` 在 `check`；
- 关闭类回调在 `close callbacks`。

需要特别注意：
- `process.nextTick` **不属于**这 6 个阶段；
- `Promise.then`、`queueMicrotask` 这类微任务也**不属于**这 6 个阶段。

它们优先级更高，通常可以先粗略记成：

> **同步代码 -> `process.nextTick` -> Promise 微任务 -> 事件循环各阶段任务**

如果要系统理解执行顺序、6 个阶段的职责、`setTimeout` 和 `setImmediate` 的差异，应继续看专题笔记：[`事件循环.md`](./事件循环.md)

## 4. Node.js 为什么适合 I/O 密集型场
Node.js 最擅长的是：
- 请求很多；
- 每个请求里有大量网络 I/O 或文件 I/O；
- 单个请求本身的 CPU 计算并不重。

典型场景：

### 4.1 BFF / API 聚合层
前端发来一个请求，Node.js 同时请求多个后端服务，再把结果组装返回。

这类场景中：
- 请求拆分多；
- 网络等待多；
- 业务编排多；
- CPU 重计算少。

这正是 Node.js 的主场。

### 4.2 实时通信
比如：
- WebSocket 聊天；
- 实时协同；
- 通知推送；
- 长连接网关。

Node.js 很适合维护大量连接，并快速响应 I/O 事件。

### 4.3 前端工程化工具
像 `Vite`、`Webpack`、`ESLint`、`Babel`、`pnpm` 周边生态，本质上都依赖 Node.js 运行时。

---

## 5. Node.js 不擅长什么
Node.js 不适合长时间霸占主线程的 **CPU 密集型任务**，比如：
- 大规模图像/视频转码；
- 复杂加密计算；
- 大型矩阵运算；
- 超长时间的同步循环；
- 非常重的 JSON 解析与数据转换。

原因很简单：

> JavaScript 主线程一旦被重计算卡住，事件循环就无法及时处理其他请求和回调。

于是你会看到：
- 定时器延迟变大；
- 接口响应抖动；
- WebSocket 心跳延迟；
- 吞吐量明显下降。

---

## 6. 遇到 CPU 密集任务怎么办
工程上常见的处理方式有 4 种。

### 6.1 任务切片

把一大段计算拆成很多小段，每处理一部分就把控制权还给事件循环，例如借助 `setImmediate()` 分批处理。

适合：

- 可以分段执行；
- 不要求一次算完；
- 希望减少主线程长时间阻塞。

### 6.2 `worker_threads`

把重计算移到工作线程里，让主线程继续负责 I/O 和调度。

适合：

- 压缩、加密、转码；
- 规则引擎；
- 数据分析；
- AI 推理前后处理。

### 6.3 多进程 / `cluster`

在多核机器上启动多个进程，让不同 CPU 核心一起处理请求。

适合：

- Web 服务横向扩展；
- 提升整体吞吐；
- 提高可用性。

### 6.4 把重任务下沉到其他服务

如果计算特别重，通常更合理的方案是：

- 让 Node.js 做网关或编排；
- 把重计算交给 Go、Java、Rust、Python 服务，或专门的任务队列系统处理。

---

## 7. 一段最适合记忆的总结

> Node.js 是一个 JavaScript 运行时。它的核心不是“多线程并行执行 JS”，而是“单线程 JS + 非阻塞 I/O + 事件循环调度”。
>
> 它非常适合 I/O 密集型应用，如 API 聚合、实时通信、前端工程化，但不适合让主线程长期承担 CPU 密集型计算。遇到重计算时，应考虑任务切片、`worker_threads`、多进程，或将任务下沉到更适合的服务中。
