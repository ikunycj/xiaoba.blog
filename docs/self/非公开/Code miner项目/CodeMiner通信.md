# Code-Miner 通信机制

> 这篇文档只回答一个问题：
>
> **Code-Miner 里，谁在和谁通信，靠什么通信，为什么这样设计？**
>
> 它不重复讲整个项目架构，而是把“通信与协作”这条线单独拎出来。

---

## 先说结论

Code-Miner 的通信不是单一协议，而是按对象分层：

| 通信对象 | 主要方式 | 作用 |
| --- | --- | --- |
| **Agent -> 模型 Provider** | OpenAI 兼容 HTTP API | 获取推理结果 |
| **Agent -> MCP 工具** | `stdio` / `sse` / `streamableHttp` | 访问外部工具和信息资产 |
| **父 Agent -> 子 Agent（普通模式）** | 进程内工具调用 + 方法返回 | 委派局部任务 |
| **父任务 -> 子任务（Batch 模式）** | SQLite 状态协同 | 任务调度、等待、恢复 |

一句话概括：

> Code-Miner 对外主要走 HTTP 和 MCP，对内普通模式走本地方法调用，Batch 模式走数据库状态协同。

---

## 1. 它没有用 A2A 吗

当前可以明确下结论：

- **没有采用 A2A（Agent-to-Agent）协议作为主通信方案**
- 父子 Agent 协作不是通过独立网络协议互相发消息
- 更不是一组自治 Agent 之间的开放式协商网络

为什么？

因为 Code-Miner 的目标不是做“多智能体开放协作平台”，而是做一套**可控、可审计、可恢复**的离线分析系统。

对这种系统来说，更重要的是：

- 调度边界是否明确
- 状态是否可持久化
- 失败后是否能恢复
- 成本与资源是否可控

所以它选择的是更工程化、更封闭、更确定的协作方式。

---

## 2. Agent 和模型怎么通信

### 主链路

Agent 与大模型 Provider 的通信主体是 **OpenAI 兼容 HTTP API**。

你可以把它理解为：

- Agent 负责组织上下文、工具结果和系统规则
- Provider 负责完成一次推理调用
- 二者之间通过标准 HTTP 请求交换消息

这层通信的特点是：

- 接口成熟
- 易于替换不同 Provider
- 易于做日志、鉴权、重试和限流

也就是说，模型服务在 Code-Miner 里被当成**推理后端**，而不是整个系统的中心。

### 为什么这里用 HTTP

因为对模型服务来说，HTTP 的收益非常直接：

- 生态成熟，兼容 OpenAI 风格接口
- 便于接入多家模型供应商
- 更容易做 API 网关、统计、审计和失败重试

这层不需要复杂长连接，也不需要 Agent 之间的协商协议。

---

## 3. Agent 和 MCP 工具怎么通信

Code-Miner 支持通过 MCP 接外部工具和信息资产。

当前笔记里明确提到的传输方式有：

- `stdio`
- `sse`
- `streamableHttp`

这几种传输的适用点不一样：

| 传输方式 | 适合场景 | 特点 |
| --- | --- | --- |
| `stdio` | 本地或同机工具 | 最简单，部署轻 |
| `sse` | 服务端持续输出结果 | 适合流式事件 |
| `streamableHttp` | 更标准的服务化接入 | 易于网关化和远程访问 |

在 Code-Miner 里，MCP 不是“锦上添花”，而是补足代码扫描无法直接获得的外部事实源，比如：

- 契约信息
- 组件资产
- 错误码
- 内部信息资产

这意味着系统并不把“所有事实都从代码里硬推出来”，而是允许通过受控协议接入权威数据源。

---

## 4. 父子 Agent 是怎么协作的

这里要分成两种模式看：

- **普通模式**
- **Batch 模式**

二者最核心的区别不在于“有没有子 Agent”，而在于**父任务如何等待和恢复**。

---

## 5. 普通模式：父子 Agent 走进程内调用
在普通模式下，父 Agent 并不会通过 gRPC、WebSocket 或 A2A 去远程拉起另一个自治 Agent。

它更像这样工作：
1. 父 Agent 在工具层决定“把这件事委派出去”
2. 调用 `delegateTask`
3. 内部创建 `SubAgent`
4. 子 Agent 执行局部任务
5. 返回结构化结果给父 Agent

所以它的本质是：
- **工具调用**
- **本地执行**
- **方法返回结果**

不是：
- 网络对话式协商
- 对等 Agent 之间的协议通信
- 松散自治节点之间的消息广播

### 这种设计的好处
1. **边界清晰**
   父 Agent 负责决策，子 Agent 负责执行，职责天然分开。
2. **失败隔离**
   单个子任务出错，不会把父 Agent 的全局状态拖坏。
3. **成本可控**
   子 Agent 生命周期短，不背长期历史。
4. **调试更容易**
   通信对象就是输入任务描述和输出结果对象，不需要额外维护协议复杂度。

### 它传的是什么
普通模式里，父子 Agent 真正交换的核心内容通常只有两类：
- `taskDescription`
- `SubAgentResult`

也就是：
- 父亲告诉孩子“你去做什么”
- 孩子回来汇报“做完了没有，结果是什么”

这比“持续对话式多智能体协商”简单得多，也更适合离线生产系统。

---

## 6. Batch 模式：父子任务走 SQLite 状态协同

Batch 模式的重点不是实时通信，而是**状态协调**。

在这种模式下，父任务和子任务不会维持持续在线的实时消息通道，而是通过数据库中的任务状态完成协作。

简化后的流程是：

1. 父任务创建子任务记录
2. 父任务状态切到 `waiting_sub_agent`
3. 子任务各自执行
4. 协调器轮询检查子任务是否都完成
5. 汇总结果后，把父任务恢复成 `in_progress`
6. 父任务继续后续执行

### 为什么这里选 SQLite

这里的 SQLite 不是拿来“存业务数据”的，而是拿来做：

- 任务状态持久化
- 并发协调
- 可恢复执行
- 父子任务同步点管理

它像一个轻量级但强一致的调度内核。

### 这种设计的关键价值

1. **可恢复**
   进程中断后，系统还能根据状态继续跑。
2. **可审计**
   每个子任务的生命周期都能查到。
3. **可并发**
   子任务完成顺序不重要，只要最终能汇总。
4. **更适合离线批处理**
   没必要为了长任务维持昂贵的实时会话。

所以 Batch 模式更像“状态机驱动的任务编排”，而不是“在线聊天式协作”。

---

## 7. 为什么普通模式和 Batch 模式不统一成一种协议
因为它们解决的根本问题不同。

### 普通模式的关注点
- 快速委派
- 局部执行
- 结果回传
- 尽量减少额外开销

### Batch 模式的关注点
- 可恢复
- 可并发
- 可持久化
- 可等待和恢复父任务

如果强行统一，很容易出现两个问题：

1. 普通模式被设计得过重
2. Batch 模式又缺少足够强的状态管理

所以 Code-Miner 选择“按场景选通信模型”，而不是追求形式上的统一。

---

## 8. 从工程视角看，这套通信设计说明了什么

### 不是“所有东西都协议化”才高级

很多人会下意识觉得：

- A2A 更先进
- 多 Agent 网络协商更高级
- 所有组件都走统一协议更优雅

但 Code-Miner 的选择说明：

> 对工程系统来说，合适比抽象更重要。

如果一个问题本质上只需要本地委派和结果返回，那就没必要引入更重的协议。

### 通信方案服务于系统目标

Code-Miner 的目标是：
- 离线
- 可恢复
- 可追踪
- 可控成本

所以它的通信方案自然偏向：
- 简单
- 可落地
- 易审计
- 易恢复

这和面向开放生态、自治协商、多租户互联的 Agent 平台不是一个问题空间。

---

## 9. 如果面试被问“Code-Miner 的通信机制”怎么答

可以用下面这套简版：

> Code-Miner 的通信是分层的。
> 对模型，主要走 OpenAI 兼容 HTTP API；
> 对外部工具，走 MCP，当前支持 `stdio`、`sse` 和 `streamableHttp`；
> 对内父子 Agent 协作，普通模式用进程内工具调用和方法返回，Batch 模式用 SQLite 做状态协同。
> 它没有采用 A2A 作为主通信方式，因为这个系统追求的是工程确定性、可恢复和可审计，而不是开放式自治协商。

如果对方继续追问，再补一句：

> 普通模式解决的是局部委派，Batch 模式解决的是长任务编排，所以二者的通信机制天然不同。

## 总结
Code-Miner 的通信机制并不追求“协议炫技”，而是围绕任务目标做最小但有效的工程选择：

- 对模型，HTTP 足够稳定
- 对外部工具，MCP 足够标准
- 对普通父子 Agent，本地调用最直接
- 对 Batch 调度，SQLite 状态协同最可靠

这恰好体现了它作为离线工程系统的设计取向：**协议服务于确定性，而不是为了抽象而抽象。**
