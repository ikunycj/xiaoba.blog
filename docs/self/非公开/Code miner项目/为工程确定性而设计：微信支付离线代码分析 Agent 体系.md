## 导语
微信支付大规模代码分析 Agent 是一套面向核心业务的离线代码智能分析基础设施，用于在无人值守、可复现、成本可控的前提下，对数万级代码仓库进行系统性理解与结构化抽取。

该系统并非面向交互式“代码问答”，而是围绕长期运行的分析任务进行工程化设计：通过将代码理解能力前移至工具与分析引擎层，结合离线 ReAct 执行模型、Todo 任务状态外化、父子 Agent 上下文隔离机制以及强一致的资源调度体系，实现对复杂代码资产的稳定分析。

目前，该系统已在微信支付内部支撑 10 万次以上离线分析任务，覆盖接口契约智能分析、调用关系强弱依赖识别等核心场景，并作为多项 AI 工程系统与研发工具的底层能力基础，持续输出可复用、可追溯的工程级分析结果。

## 背景与问题定义

### 背景
微信支付内部沉淀了数万级代码仓库，覆盖核心支付链路与大量长期演进的一线业务逻辑。接口契约、协议定义、业务用例、错误码以及关键工程约定，主要以代码实现的形式分散存在于各仓库之中，缺乏统一、结构化的抽取与表达机制。这使得 MIS 系统、自动化测试体系以及研发工具，难以在规模、一致性与时效性三个维度上对这些业务知识进行持续理解和复用。

在实际工程中，设计文档与业务说明往往滞后于实现，配置与策略分散在多个系统，唯有代码能够完整反映当前系统的真实行为和边界条件。因此，在 AI 时代，代码事实上成为最具权威性和实时性的业务知识来源。但如果这些代码资产只能以“文本”或“文件”的形式被访问，而无法被系统性理解、抽象与组织，AI 的能力将只能停留在局部代码补全与单点问答层面，难以参与到接口契约治理、调用关系分析、风险评估与自动化决策等核心工程场景中。

要使 AI 真正进入核心业务，就必须将分散在大规模代码仓库中的隐性业务知识，转化为可被持续分析、组合与复用的结构化工程资产。

### 核心工程约束
如果系统仍将对代码的理解完全依赖大模型的上下文输入，随着分析规模和复杂度的持续扩大，必然会暴露出三类结构性且不可规避的问题：

1. **上下文容量瓶颈**：大模型的上下文窗口无法长期承载完整的工程结构与跨仓库依赖关系，分析过程只能在局部信息下进行，导致对整体系统行为的理解天然不完整。
2. **推理稳定性不足**：上下文选择具有强不确定性，只要关键代码、契约或隐含约定未被纳入输入，模型的推理结论就会发生明显漂移，难以保证分析结果在多次执行中的一致性和可复现性。
3. **成本与吞吐不可控**：在缺乏知识外化与复用机制的情况下，每一次分析都需要重复加载大量领域与工程背景信息，随着任务规模放大，推理成本和系统吞吐将呈非线性增长，难以进行工程级预算和容量规划。

在微信支付这样对稳定性、可预测性和风险可控性有极高要求的核心金融业务环境中，上述不确定性在工程层面是不可接受的。

### 首要设计判断
大规模代码分析，不能建立在“模型反复在上下文中重新理解代码”的前提之上。

系统必须在架构层面重新划分“**理解**”与“**推理**”的职责。

## 总体设计理念
### 能力前移：从上下文理解到工具化理解
系统将可程序化的代码理解前移到工具与分析引擎层，由专用分析工具完成：
- 代码解析
- 结构抽取
- 协议识别
- 规则化裁剪

语言模型不再承担底层事实理解职责，而是基于这些高密度、结构化的分析结果，进行组合、决策与抽象推理。

在这一架构下：
- 工具是上下文压缩器与领域知识的长期载体。
- 模型是短期推理与策略执行单元。

### 执行模型的根本变化
微信支付代码分析 Agent 不再以“对话是否结束”作为运行终点，而是以分析任务是否完成作为唯一终止条件。

通过离线化的 ReAct 执行模型与 Todo 任务机制，系统将以下信息从自然语言对话中彻底外化，形成独立、可持久化的任务状态层：
- 要分析什么
- 当前进度如何
即使对话历史被压缩、截断或重写，分析任务结构与执行进度仍然可以被完整恢复。

### 工程化目标
该系统的目标不是构建一个“更聪明的对话式助手”，而是一套面向核心业务、以工程确定性为优先级、可长期运行、可控成本、可复现的离线代码智能分析基础设施。

## Agent 工程架构设计

### 架构定位
微信支付代码分析 Agent 采用面向任务的工程架构，而非对话交互架构，核心目标是支撑大规模、离线、系统性的代码智能分析任务。

### 离线 ReAct 执行模型
Agent 基于 ReAct（Think → Act → Observe）循环执行，但引入关键工程化调整：

- 以任务完成状态作为唯一终止条件

```ts
if (!thoughtResult.action) {
  if (await this.hasPendingTodos()) {
    continue;
  }
  this.status = AgentStatus.COMPLETED;
  break;
}
```

Agent 不会因“当前轮次没有动作”而退出，只要 TodoList 中仍存在未完成任务，就会继续执行。

### 父子 Agent 协作机制
在 CodeMiner 中，Agent 并非以单体形式运行，而是采用 Planner / Worker（父子 Agent）模式，用于支撑大规模、长期、无人值守的代码分析任务。

父 Agent（Planner）的职责是全局性的。
	它不参与任何具体代码细节的推理，而是专注于：
	- 拆解整体分析目标，生成可执行的子任务结构
	- 维护 Todo 列表，跟踪任务完成度与依赖关系
	- 控制任务推进节奏，决定并行度、重试与回溯策略

在这一模式下，Planner 关注的是“做什么、做到哪一步、是否可以继续”，而不是“如何分析某一段代码”。

子 Agent（Worker）则是执行单元。
	每个 Worker 只围绕一个明确、有限的分析对象运行，具备短期、一次性的上下文，其职责包括：
	- 执行具体的代码或协议分析任务
	- 调用受控工具产出 Artifact
	- 将结果以结构化 Report 的形式返回给 Planner

Worker 的生命周期与任务绑定，任务完成即退出，不继承历史上下文，也不承担跨任务记忆责任。

这种 Planner / Worker 模式带来的核心价值在于：

- **任务可拆解**：复杂分析被分解为若干独立子任务，避免单一上下文持续膨胀。
- **并行可控**：Planner 根据资源调度结果与依赖关系，决定 Worker 的并发数量，而不是简单地“同时启动”。
- **故障隔离**：单个 Worker 的失败不会污染全局上下文，Planner 可以选择重试、回滚或调整策略，系统整体保持稳定。

从工程角度看，父子 Agent 的差异并非能力层级，而是上下文责任与生命周期的不同：

- Planner 持有长期、稳定的任务状态
- Worker 承担短期、局部的推理负载

这一模式使 Agent 系统从“对话驱动”演进为“任务驱动”，使代码分析不再依赖连续上下文的幸运命中，而是建立在显式规划、可调度执行与可恢复状态之上。

在长期、离线、无人值守的分析场景中，这种 Planner / Worker 模式是 Agent 能够稳定运行的必要结构，并不仅仅是并发优化手段。

## 代码分析工具体系设计

### 工具的工程定位

在微信支付代码分析 Agent 中，工具并不是“能力插件”或“可选增强”，而是整个系统得以成立的前提条件。

如果没有工具层的存在，Agent 将被迫把“理解代码、筛选信息、构造结构”这些工作全部压到模型上下文与推理过程中，系统在规模、稳定性与成本上都会迅速失效。

工具的第一职责，是在模型推理之前完成一次信息蒸馏。

真实工程中的代码资产具有典型特征：

- 体量巨大（跨仓、跨模块、跨语言）
- 信息密度极低（大量样板代码、框架代码、重复逻辑）
- 对推理真正有价值的信号高度稀疏

如果直接将原始代码送入模型，模型消耗的大部分 Token 实际上用于“无意义读取”，而非推理本身。工具层的作用，正是将这一步从模型中前移到系统中。

工具负责将“高体积、低信息密度”的数据压缩为结构化结果。

例如：

- 从成千上万行代码中提取接口定义、调用关系、错误码
- 将隐含在实现中的依赖关系转化为图结构或摘要
- 将跨文件、跨仓的事实收敛为有限、可验证的分析输入

模型看到的，不再是“代码原文”，而是已经被工程规则过滤和整理过的高价值信息。

同时，工具承担着“固化高频、稳定领域知识”的职责。

在代码分析场景中，大量知识是：

- 稳定的（框架约定、接口规范、工程惯例）
- 高频复用的（每次分析都会用到）
- 不适合反复占用上下文推理

通过将这些知识编码进工具的行为与输出结构中，可以：

- 减少 Prompt 中的显式说明
- 降低模型对上下文的依赖
- 提升推理一致性与可复现性

从这个角度看，工具并不是“帮模型做事”，而是在替模型承担不该由模型承担的工作。

### 安全性与边界控制
在金融级工程环境中，分析系统的首要前提不是“强”，而是“不破坏”。

因此，代码分析 Agent 在设计之初就必须明确安全边界，而这些边界同样由工具体系来强制保证。

Agent 对源代码仓库始终保持只读权限。

所有分析工具仅允许：
- 读取代码与元数据
- 构建中间表示（AST、依赖图、索引等）
- 生成分析结论

Agent 无法、也不应对任何真实业务代码产生写入行为。

所有写操作被严格限制在分析目录或临时目录中。

例如：

- `/data/ai_analysis`
- `/tmp/codeminer`

这些目录与真实工程代码物理隔离，生命周期可控、可清理、可审计，确保任何中间产物、缓存或结果文件都不会反向影响工程事实。

这种边界设计的核心目的，是保证分析过程不污染工程事实。

代码分析 Agent 的职责是：

- 观察
- 抽象
- 推断

而不是：

- 修改
- 补全
- “顺手修一下”

一旦分析系统具备写入生产代码的能力，其输出将不再是“结论”，而是“变更”，系统风险性质也会发生根本变化。

因此，工具体系不仅是能力载体，更是安全与责任边界的执行者。

通过工具级别的权限控制与行为约束，系统可以在不信任模型推理结果的前提下，依然保证：

- 工程资产不被污染
- 分析过程可回溯、可审计
- Agent 行为始终处于可控范围内

### 工具集设计
在微信支付代码分析 Agent 中，工具集并非“通用能力集合”，而是一组围绕上下文工程、安全边界与分析确定性精心裁剪的工程工具。其设计原则不是“能做什么”，而是哪些事情必须在模型推理之外、以确定性方式完成。

#### 1. 基础工具：受控的搜索、读取与写入
基础工具并非简单复用 IDE 或操作系统能力，而是在性能、安全与上下文成本约束下的工程化版本。

**搜索（Grep）**

搜索能力在 QQMail 层级进行了明确限制，避免在超大仓库或全量历史中进行无界扫描。

该限制并非功能缺失，而是基于工程现实的取舍：无约束的全局搜索在离线、多 Agent 并发场景下会迅速成为性能瓶颈，并拖垮整体吞吐。

**读文件（ReadFile）**

文件读取以“可进入上下文”为设计前提，严格限制单次可读行数，并在执行前进行 Token 预估。

当预期窗口超过安全阈值时，工具会直接拒绝执行，而不是将风险转嫁给模型上下文。这一机制确保：上下文超载是工具层问题，而不是推理层事故。

**写文件（WriteFile）**

写操作仅允许发生在分析目录或临时目录中，用于保存 Artifact、分析结果与中间状态。工具在权限层面强制保证分析过程不污染任何工程事实，使 Agent 始终处于“观察与推断”角色，而非“修改者”。

这些基础工具的共同特征是：功能受限但行为确定，其目标是为上层分析提供安全、可预测的输入，而不是最大化操作自由度。

#### 2. 专用分析工具：XWI / XCGI 框架解析

XWI、XCGI 等框架大量依赖 Proto / JSON 协议定义，具有结构复杂、上下文消耗极高的特点。如果以原始文本形式进入模型，上下文成本将呈指数级上升。

为此，系统提供了专用框架分析工具，能够：

- 直接解析 Proto / JSON 结构
- 提取接口、字段、约束与调用关系
- 以结构化结果输出分析要点

通过工具侧解析，模型不再承担“读懂协议格式”的负担，而只需基于已压缩的语义结果进行推理。这一步本质上是将高频、稳定的框架知识前移固化，显著降低上下文占用与推理不确定性。

#### 3. 专用分析工具：AST / CST 分析（基于 tree-sitter）

针对源代码结构理解，系统引入了基于 tree-sitter 的 AST / CST 分析工具，用于完成模型难以稳定完成的结构性任务。

该工具支持：

- 多语言 AST / CST 构建
- 基于 CST 的模式匹配（Pattern Matching）
- 准确定位函数、调用、条件分支与语法结构

其价值不在于“更快”，而在于确定性：结构匹配规则一旦定义，其结果不依赖上下文、不随推理波动，是可复现、可验证的工程事实。模型仅基于这些事实进行高层语义判断，从而避免“猜代码结构”的不稳定行为。

#### 4. 专用分析工具：HAR 包分析工具

在涉及网络调用、接口行为与线上问题回溯的场景中，系统提供了针对 HAR（HTTP Archive）包的专用分析工具。

该工具针对抓包分析场景进行定制，能够：

- 解析请求 / 响应结构
- 还原调用链与关键参数
- 抽取异常模式与关键行为特征

相比将 HAR 原文交由模型分析，这种工具化处理可以将大量噪声数据压缩为少量可推理事实，使 Agent 能在极低上下文成本下完成复杂的网络行为分析。

#### 5. 专用分析工具：XSOA 资产 MCP

此外，系统还挂载了微信支付 XSOA 信息资产数据的 MCP，用于解决“微信支付内部信息资产获取”这一类不适合通过代码扫描或文本分析完成的问题。

在真实工程中，大量关键信息并不直接存在于代码中，而是以资产化、平台化的形式存在，例如：

- 调用关系
- 用例
- 子域
- 系统等 CMDB 数据

如果仅依赖代码侧分析，Agent 要么无法获取这些信息，要么被迫通过不稳定的上下文推断来“猜测事实”，从而引入不确定性。

通过 XSOA MCP，Agent 可以以受控、结构化、只读的方式直接获取这些信息资产数据，并将其作为权威事实源使用。

#### 6. 数据修复工具：JSON Util

在 CodeMiner 的分析流程中，结构化数据是系统能够持续运行与复用分析结果的前提。然而在实际工程中，即便 Prompt 设计再严格，LLM 生成的 JSON 结果仍不可避免地会出现格式层面的不可靠性，例如字段缺失、类型错误、语法不完整或嵌套结构不闭合等问题。

如果将这些问题直接暴露给下游系统或依赖人工修复，将导致两个不可接受的后果：

- 一是分析流程在无人值守条件下频繁中断
- 二是结构化结果的可信度无法保证

因此，CodeMiner 在工具体系中引入了专用的数据修复工具：JSON Util，用于在模型输出与系统消费之间，构建一道确定性的工程防线。

JSON Util 的职责并不是“猜测模型意图”，而是严格围绕结构有效性展开，具体包括三个阶段：

1. **格式校验**  
   对 LLM 输出进行严格的 JSON 语法与 Schema 校验，明确区分“语义可能不完整”与“结构不可解析”两类问题，避免模糊失败。

2. **自动修复**  
   对于可判定的常见结构错误（如缺失括号、非法转义、字段类型偏差等），工具会在不引入新语义的前提下进行自动修复，确保修复过程是确定性且可复现的。

3. **失败归因与反馈引导**  
   当自动修复不可行时，JSON Util 会输出明确、结构化的失败原因，并将其作为反馈注入下一轮 Agent 推理，引导 LLM 针对具体错误进行修正，而非进行盲目重试。

通过这一机制，系统将“JSON 是否可用”的判断权，从模型的隐式行为中剥离出来，转化为工具层的显式约束。

模型只负责生成候选结果，而结果是否满足系统要求，由工具给出确定性裁决。

整体而言，这套工具集的设计目标并非“覆盖所有操作”，而是：

- 在模型推理之前完成信息蒸馏
- 将高体积、低确定性的输入转化为结构化事实
- 通过权限与约束设计，确保分析过程安全、可控、可复现

工具不是 Agent 的外挂，而是上下文工程与系统可靠性的第一道防线。

## 上下文工程

### 提示词工程
在等价语义与约束强度下，英文 Prompt 的 Token 占用显著低于中文 Prompt，这一差异并非表层表达习惯所致，而是由大模型底层分词机制与工程运行特性共同决定的，因而在长程、规模化分析场景中具有明确的工程优势。

#### 手搓的 Token 分析工具

第一，分词机制决定了基础 Token 成本差异。

主流大模型采用的 BPE / Unigram 等子词分词器，主要以英文语料为核心进行优化。英文具备天然的词边界（空格）与稳定的词形结构，一个完整语义单元往往可被映射为 1–2 个 Token；而中文缺乏显式分隔符，单个汉字通常对应 1 个 Token，常见的技术概念、抽象名词往往需要 3–6 个 Token 才能完整表达。在等价语义下，中文 Prompt 的 Token 密度天然更高。

第二，约束型表达在英文中更具“压缩性”。

在代码分析、规则约束、角色限定等 Prompt 中，英文可以通过固定句式与函数式表达（must only never strictly under condition that ...）在极少 Token 内完成高强度约束；而中文往往需要通过补充状语、强调语气或重复限定来避免歧义，导致在保持同等约束强度时 Token 数量进一步膨胀。

第三，模型训练分布使英文 Prompt 的语义密度更高。

大模型在英文技术语料、代码注释、规范文档上的训练密度显著高于中文，导致英文 Prompt 在较短上下文中即可触发稳定、明确的推理模式；而中文 Prompt 若压缩过度，更容易出现歧义或推理漂移，工程上往往需要“补 Token 保稳定”，进一步放大 Token 成本差距。

第四，在长程与规模化场景中，Token 差异会被线性放大。

在离线代码分析、跨仓扫描、父子 Agent 调度等场景中，Prompt 会被：

- 重复加载（每个子任务一次）
- 叠加记忆摘要、工具描述、上下文状态
- 放大为数万到数十万次调用

此时，即便单次 Prompt 节省 20%–40% 的 Token，占用差异也会直接转化为：

- 更低的上下文溢出风险
- 更稳定的推理结果
- 更可控的成本与吞吐

在金融级、无人值守的分析系统中，这种确定性差异具有明确的工程价值。

因此，选择英文作为核心 Prompt 表达语言，并非语言偏好问题，而是上下文工程与系统可扩展性的必然选择。

中文更适合作为结果解释与人类交互层，而英文 Prompt 更适合作为 Agent 的“指令语言”和“约束语言”，承载长期、稳定、可规模化的推理负载。

### KV 缓存

在 CodeMiner 的大规模离线分析场景中，模型推理并非孤立调用，而是由大量结构相同、仅在局部行为上发生变化的 Agent 推理请求构成。如果每一次推理都完整重放系统提示词与全量指令，不仅会造成巨大的 Token 浪费，还会引入不必要的推理波动。

因此，CodeMiner 将 KV 缓存视为系统级优化手段，而非模型层面的偶然收益，并围绕这一目标，对提示词结构与 Agent 行为注入方式进行了专门设计。

核心原则是：系统提示词保持稳定，Agent 行为以增量方式插入。

系统提示词承载的是长期稳定、不随任务变化的规则性信息，包括：

- Agent 的角色定义与行为边界
- 工具使用规范与安全约束
- 上下文工程、Todo、Artifact、父子 Agent 通信等核心机制约定

这些内容在整个分析生命周期中高度稳定，适合作为 KV 缓存的固定前缀。一旦该前缀在推理服务中被缓存，其对应的 Key / Value 状态即可在后续请求中被直接复用，避免反复计算。

### 工具并发化

在实际工程环境中，Agent 的整体性能瓶颈主要来自工具执行阶段，而非模型推理本身。这一结论在单次交互场景中并不明显，但在长程、离线、规模化分析任务下，会迅速成为决定系统上限的核心因素。

首先，模型推理的耗时具备相对稳定且可预期的上界。

在现代推理服务中，单次 LLM 调用的延迟主要受模型规模与上下文长度影响，其波动范围相对可控；而且推理过程本身是高度并行、资源可弹性扩展的，工程上可以通过批处理、模型复用、异步调用等方式获得近似线性的吞吐提升。因此，在规模化场景中，模型推理更像一个“可定价、可扩容”的计算问题。

相比之下，工具执行是不可忽视的真实瓶颈来源。

Agent 在执行代码分析任务时，绝大多数时间消耗在工具侧，包括但不限于：

- 跨仓库文件扫描与 I/O 读取
- AST / IR 构建与多语言解析
- 依赖图、调用图的生成与遍历
- 外部系统访问（代码搜索、索引、契约平台、元数据服务等）

这些操作具有明显的 I/O 密集、异构计算、长尾分布特征，执行时间往往比模型推理高一个数量级，并且单次失败或阻塞就会放大整体任务时延。

其次，串行工具执行会直接导致 Agent 吞吐呈指数级下降。

在典型的分析任务中，一个 Agent 往往需要调用数十甚至上百次工具。如果这些工具按照“思考 → 调用 → 等待 → 再思考”的串行模式执行，那么系统的总耗时将近似等于所有工具执行时间的线性叠加，Agent 实际上退化为一个“慢速编排器”。

在这种架构下，哪怕模型推理时间被优化到极致，整体吞吐仍然被工具执行所锁死，系统无法支撑任何真实规模。

因此，并发执行并非性能优化项，而是系统可行性的前提条件。

并发化的目标不是“更快一点”，而是：

- 将 Agent 从串行阻塞模型升级为并行调度模型
- 让 I/O 等待、计算密集任务与推理阶段充分重叠
- 将单任务耗时从“工具总时长之和”压缩为“关键路径长度”

只有在并发执行成立的前提下，Agent 才可能具备稳定的吞吐曲线，支撑跨仓、跨模块、长时间运行的分析任务。

最后，并发能力决定了 Agent 是否具备工程级扩展性。

在大规模代码分析中，系统关注的不再是“单个任务有多快”，而是：

- 单位时间内可以完成多少分析
- 任务规模扩大 10 倍时，资源与时延是否可线性扩展
- 是否存在不可控的长尾任务拖垮整体系统

这些问题的答案，几乎完全取决于 Agent 是否在工具层面实现了有效的并发调度与资源隔离，而非模型本身的推理速度。

### TODO：任务状态外化机制
Todo 机制的核心目标，是将 “要做什么、做到哪一步” 从不稳定、易衰减的对话上下文中剥离出来，形成一个独立、显式、可持久化的任务状态层。

它解决的不是“记忆多不多”的问题，而是 Agent 是否还能持续推进一个长期任务 的问题。

上下文可以被压缩，但任务不能被遗忘。

在长程代码分析场景中，对话历史不可避免地会被裁剪、摘要或替换；而一旦“剩余任务”仅存在于上下文中，任何一次压缩、漂移或重启，都会直接导致任务中断或方向偏移。这种不确定性在无人值守的工程系统中是不可接受的。

Todo 机制通过“任务状态外化”，将记忆责任从模型转移到系统。

具体而言：

- 对话上下文仅承载即时推理所需的信息
- TodoList 承载长期目标、子任务拆分与完成状态
- 任务进度以结构化形式存储，可持久化、可回放、可校验

这样，Agent 的连续性不再依赖模型是否“记得”，而依赖系统是否“记录”。

在每一轮 ReAct 循环结束后，系统都会对未完成任务进行显式回注。

这一过程并不是简单的重复提醒，而是：

- 从 TodoList 中筛选未完成或部分完成的任务
- 将其以“当前必须继续推进的事实”形式注入下一轮推理
- 强制 Agent 在思考阶段重新对齐任务边界与优先级

从工程角度看，这一步相当于为 Agent 提供一个稳定的“任务锚点”，防止其在多轮工具调用与上下文演化中逐步偏离原始目标。

Todo 机制使 Agent 从“对话驱动”升级为“状态驱动”。

没有 Todo 的 Agent，本质上是依赖上下文连续性的对话系统；

引入 Todo 后，Agent 才真正具备了：

- 跨轮次、跨上下文的任务连续性
- 可中断、可恢复、可重试的执行能力
- 面向长时间运行任务的工程可靠性

在规模化分析场景中，Todo 不是辅助功能，而是系统存活条件。

当分析任务跨越数小时、数千仓库、数万工具调用时，唯一可信的“记忆”只能来自显式的任务状态，而不是隐含在 Token 流中的上下文痕迹。

因此，Todo 机制的本质，是把“不要忘记要做什么”这件事，从大模型的能力问题，转化为系统的工程保证。

这也是代码分析 Agent 能够长期运行、稳定收敛、可规模化扩展的关键基础之一。

### 滑动窗口 + 摘要总结

在长程代码分析 Agent 中，上下文管理的目标并不是“尽可能记住更多内容”，而是在有限上下文容量内，最大化推理稳定性与任务连续性。

滑动窗口与摘要总结的组合，正是为此设计的一套分层记忆机制。

滑动窗口用于保留近期的高保真推理过程。

滑动窗口会完整保留最近若干轮对话与工具调用结果，包括：

- 当前分析假设与中间推理路径
- 刚完成的工具执行反馈
- 尚未被验证或收敛的推断

这些信息具有高度时效性，对下一步推理至关重要，一旦被压缩，极易导致推理断裂或错误跳转。因此，滑动窗口的职责是不压缩、不解释、不抽象，只保真，确保 Agent 在“当前问题空间”内连续思考。

摘要总结用于保存已确认的结论与稳定状态。

当某一阶段的分析结果已经完成验证、形成确定结论后，其原始推理过程本身的价值迅速下降，而结论与状态才是长期有用的信息。摘要机制会：

- 将已确认的事实、结论、依赖关系进行结构化压缩
- 丢弃冗余的推理痕迹与重复上下文
- 以低 Token 成本，长期保留高价值信息

从工程角度看，摘要承担的是“知识固化”的职责，而不是推理本身。

两者形成明确分工，而非互相替代。

- 滑动窗口解决的是“我现在在想什么”
- 摘要解决的是“我已经确认了什么”

如果仅依赖滑动窗口，上下文必然随时间溢出；如果过早依赖摘要，推理会在未收敛前被强行压缩，导致结论漂移。

与 Todo 机制的边界必须清晰，避免职责混淆。

- 滑动窗口：承载短期推理与即时上下文
- 摘要总结：承载长期稳定结论与分析状态
- Todo 机制：承载尚未完成的任务与执行计划

摘要不记录“还要做什么”，Todo 不存储“已经想过的细节”。

这种分工避免了两个常见问题：

- 任务丢失：上下文压缩后，未完成任务随推理一起消失
- 重复分析：已完成结论未被固化，Agent 在后续轮次反复推导

滑动窗口 + 摘要 + Todo，共同构成 Agent 的三层记忆体系。

它们分别对应：

- **思考中的工作记忆**
- **已确认的长期记忆**
- **可执行的任务记忆**

在规模化代码分析场景中，只有当这三层职责清晰、相互解耦，Agent 才能在上下文不断变化的情况下，持续、稳定、可验证地推进复杂任务。

因此，该机制并非为了“更聪明地记忆”，而是为了让 Agent 在不可避免的上下文压缩中，依然保持工程级的可靠性与可控性。

### 父子 Agent 协作：上下文作用域的工程化切分

在微信支付大规模代码分析场景中，上下文工程要解决的核心问题并不是 “如何让模型记住更多内容”，而是 如何防止无关、低价值或已完成的信息持续占用上下文空间，并对后续推理造成污染。

在长期、离线、无人值守的任务中，单一 Agent 持续累积上下文将不可避免地导致上下文膨胀、无关信息占位以及一次裁剪失误引发的整体推理漂移，因此，上下文工程必须具备 “作用域” 概念，而不仅是简单的长度控制。
在长期、离线、无人值守的任务中，单一 Agent 持续累积上下文将不可避免地导致上下文膨胀、无关信息占位以及一次裁剪失误引发的整体推理漂移，因此，上下文工程必须具备“作用域”概念，而不仅是简单的长度控制。

父子 Agent 的引入，本质不是调度模式，而是一种上下文隔离机制。

父 Agent 只维护全局分析目标、Todo 任务结构以及子任务状态与依赖关系，不承载任何具体代码细节；子 Agent 则拥有短期、一次性的局部上下文，仅围绕当前分析对象推理，任务完成即退出，不累积历史负担。

二者的根本差异不在能力，而在上下文责任范围的严格划分。

为了进一步降低上下文负载，系统明确区分两类信息流：Artifact 与 Report。
- Artifact 承载代码片段、AST、Schema 和结构化分析结果，体积大但信息密度高，通过持久化存储并以指针方式引用，永远不进入父 Agent 上下文；
- Report 仅包含任务执行状态、关键结论摘要和 Artifact 指针，体积小、结构稳定，仅用于调度与决策。

在此基础上，引入横向流动（Lateral Flow）：
- 子 Agent 之间通过 Artifact 指针直接共享分析结果，父 Agent 只维护任务依赖与状态变化，避免成为上下文汇聚点，从而防止上下文级联污染，确保每个子 Agent 的上下文始终保持局部、短期、可控。

从上下文工程角度看，父子 Agent 机制带来的工程结论是明确的：上下文被隔离、被去中心化、且具备可恢复性。

任务状态保存在 Todo 中，分析事实保存在 Artifact 中，而上下文仅作为短期执行介质存在；即使发生摘要、截断或 Agent 重启，系统仍可从 Todo + Artifact 中恢复完整分析进度。

## 资源调度机制设计

在多 Agent、多进程并行运行的离线代码分析系统中，资源调度之所以成为系统级问题，是因为模型不再是“随用随取”的函数调用，而是一种受限、稀缺、且具有外部约束的共享资源。

Provider / API Key 天然存在并发与限流上限，不同 Model 在成本、延迟与分析价值上差异显著，而 Agent 与子 Agent 的数量又随任务拆分动态变化；在这种条件下，如果缺乏统一、强一致的调度机制，系统必然演化为并发超限、级联重试、资源泄漏与成本失控的混乱状态。

因此，资源调度的目标并不是“尽量用满”，而是在并发、成本与稳定性之间建立可证明的工程约束。这要求调度机制具备多进程并发安全、强一致性、异常可恢复能力，同时又必须保持低延迟、可扩展，并且在工程上可观测、可调试。

在这一规模与复杂度下，基于文件锁的方案并不可持续。文件锁的粒度过粗，锁持有时间难以控制；进程异常退出极易留下孤立锁；更重要的是，锁状态不可查询、不可审计，调度系统退化为“黑盒”，一旦出现问题几乎无法定位与恢复。这类方案在并发规模提升后，只会放大系统不确定性。

选择 SQLite 作为调度内核，并非因为它是“数据库”，而是因为它在这里充当的是一个嵌入式、强一致的并发状态协调器。在 WAL 模式下，SQLite 支持多进程并发读写；ACID 事务为并发安全提供硬保证；busy_timeout 机制可以平滑处理锁等待；单文件形态与零部署成本，又将运维复杂度压到最低。这是在当前问题规模下，在复杂度与可靠性之间的最优工程平衡。

并发调度的关键在于一个不可拆分的原子过程：检查当前并发数 → 判断是否可用 → 更新占用状态。

这一步如果不是在事务中完成，任何并发条件下都会产生竞态，因此事务并不是“实现细节”，而是调度机制成立的前提。

在调度模型上，系统采用 Provider + Model 两级并发控制。

Provider 级约束来自外部服务的真实限制，Model 级约束则反映系统内部的策略选择与成本偏好；二者不可合并，否则将失去精细化调度与降级空间，一旦某个 Provider 或 Model 不可用，系统将无法做出局部、可控的调整。

其中，active_sessions 并非冗余字段，而是资源占用的唯一事实来源。它既是并发判断的依据，也是探活与回收机制的锚点；没有这一显式状态，系统将无法区分“正在使用”“异常遗留”与“已释放”的资源占用。

考虑到 Agent 进程可能异常退出，调度系统必须覆盖完整的资源生命周期，因此引入了基于 PID 的探活机制（ProbeService）与异步回收机制（CleanupService）。二者共同构成闭环，确保任何失联进程都不会永久占用并发额度，从而避免“幽灵并发”长期侵蚀系统吞吐。

在这种设计下，SQLite 调度机制为 Agent 提供的是一个全局可见、强一致的资源事实源。

Agent 不需要猜测并发是否可用，也不需要通过重试“碰运气”，而是基于调度结果明确调整执行策略、延迟或拆分任务。资源约束被前移为系统级事实，而非隐含在 Agent 的推理或经验判断之中。

最终，资源调度不再是“调用前的一段代码”，而是支撑多 Agent 离线分析系统可规模化运行的基础设施；它与 Agent 执行模型协同工作，确保系统在并发、异常与规模增长下，依然保持可控、可解释与可恢复。

- bounded cost
- controlled scope

## Must
- Focus on task completion, not conversation.
- Use tools to obtain facts instead of guessing.
- Track progress using TodoList (persistent task state).
- Prefer structured, high-density outputs.
- Avoid unnecessary explanations.

## Must NOT
- Ask the user questions.
- Assume facts without tool evidence.
- Modify source repositories.
- Expand task scope without explicit task requirement.

## Output Style
Keep responses concise and direct.
Responses (excluding code blocks) must not exceed 4 lines.
Use Markdown when appropriate.

## Tooling Principle
Tools are the primary source of truth.
The model is a short-horizon reasoning and strategy unit.

## Core Performance Principle: Prioritize Parallel Execution
Before each iteration, always ask:
"Can I call multiple tools in parallel?"
Only execute tools sequentially if strict dependency exists.

## Sub-Agent Delegation Policy
Use sub-agents only when:
1) Tasks are completely independent, and
2) Parallel execution can significantly reduce total latency.

Do NOT delegate decision-making responsibilities to sub-agents.
Sub-agents are executors, not planners.

## Termination Condition
The agent must NOT stop because no action is produced in a single iteration.

- TodoList has no pending items, AND

## 附录 A：为什么不采用现成 Code Agent

本附录不讨论模型能力或交互体验，仅从**工程可控性**出发，说明为什么 Claude Code 等现成 Code Agent 不适配微信支付的大规模离线代码分析场景。

微信支付代码分析 Agent 是一套**长期运行、离线执行、无人值守、面向核心业务**的工程系统，其首要目标是**稳定性、可复现性与成本可控**。因此，系统行为、资源使用与分析路径必须是**可定制、可约束、可调度**的。

现成 Code Agent 的首要问题在于不可定制。

其定制能力主要停留在 Prompt 与少量工具开关层面，而在本场景中，真正需要定制的是 Agent 的内核行为：任务拆解与回溯策略、上下文生命周期、子任务隔离模型、工具语义与权限边界等。这些能力决定的是**系统如何运行**，而非“如何提问模型”。当定制能力止步于 Prompt 层时，Agent 在工程上仍是黑盒。

进一步地，现成 Code Agent 无法定义“任务完成”的工程语义。

其终止条件通常依赖模型判断“是否完成”，而在代码分析系统中，“完成”必须由工程规则判定：覆盖范围是否完整、跨仓依赖是否穷尽、结论是否存在冲突。这些状态只能通过 Todo、Artifact 与覆盖率规则**显式管理**，而不能交由模型自行判断。

第二个结构性问题是资源无法充分调度。

现成 Code Agent 通常假设模型调用是即时且无需协调的，但在真实工程中，模型是稀缺资源：Provider 与 API Key 存在并发上限，不同模型成本差异巨大，子任务数量动态变化，失败重试会被级联放大。没有**全局一致的资源事实源**，就无法做 Provider / Model 两级限流，也无法避免并发风暴与成本失控。

在微信支付代码分析 Agent 中，模型调用必须经过**强一致调度内核（SQLite）**的事务判定，Agent 的执行策略需基于实时资源状态调整。资源调度是系统级问题，而不是 Agent 内部的优化项，现成 Code Agent 既不提供，也无法外接这一能力。

第三个问题在于工具体系并非为“代码分析”而设计。

现成 Code Agent 的工具目标是“辅助写代码”，提供文件操作、搜索或简单查询；而代码分析所需的工具必须承担**上下文压缩与知识蒸馏**的职责，直接输出结构化语义，内建领域规则与工程约定，具备严格的只读边界，且结果可复用、可存档、可被下游系统消费。
