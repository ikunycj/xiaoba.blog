## 一句话定义

Code-Miner 是一套面向微信支付场景的**离线代码分析 Agent 基础设施**，目标不是回答一次性的代码问题，而是在无人值守、可恢复、可控成本的前提下，对大规模代码仓库持续做结构化分析。

它更接近：
- 一个“任务驱动”的代码智能分析系统
- 一个“工程化”的 Agent 运行框架
- 一个把代码资产转成结构化事实的离线生产线

而不是：
- 面向即时对话的聊天机器人
- 靠一次长上下文硬吃整个仓库的代码问答工具
- 单轮推理后直接给结论的轻量 Copilot

---

## 它到底解决什么问题
微信支付内部沉淀了大量代码仓库、协议定义、错误码、接口契约和工程约定。真正的问题不在于“代码能不能被看见”，而在于这些知识长期以分散代码的形式存在，难以被稳定抽取、复用和追踪。

如果完全依赖模型临时阅读上下文，会很快遇到三类问题：

1. **上下文装不下**
   单次推理无法稳定覆盖跨仓库、跨模块、跨协议的完整事实链路。
2. **结论不稳定**
   只要输入切片稍有变化，推理路径和输出结论就可能漂移。
3. **成本不可控**
   每次都重新把背景、规则、代码片段塞进上下文，吞吐和预算都会失控。

所以 Code-Miner 的核心思路不是“让模型记住更多”，而是：
- 让工具先做代码理解
- 让系统外化任务状态
- 让模型只承担短期推理和策略决策

## 核心设计判断
### 1. 从“对话驱动”改成“任务驱动”
Code-Miner 不以“当前对话是否结束”作为终止条件，而以“任务是否真正完成”为准。

这意味着：

- 系统会持续检查 Todo 状态
- 一轮没有动作，不代表任务结束
- 只要还有未完成目标，Agent 就继续推进

这类设计特别适合长期离线分析，而不适合传统问答 UI。

### 2. 从“上下文理解”改成“工具化理解”
Code-Miner 把大量可程序化的理解工作前移到工具层，比如：
- 文件读取和搜索
- AST/结构解析
- 协议识别
- 结果裁剪和结构化抽取

这样模型拿到的不是原始代码洪水，而是经过压缩后的高密度事实。
### 3. 从“单 Agent 长跑”改成“Planner / Worker”
Code-Miner 不让一个 Agent 长时间背着所有上下文做完整分析，而是拆成：

- **父 Agent / Planner**
  负责拆任务、维护 Todo、控制推进节奏
- **子 Agent / Worker**
  负责围绕单个局部目标执行分析并返回结果

这本质上不是并发技巧，而是**上下文责任的工程化切分**。

## 系统结构怎么看
可以把 Code-Miner 粗略理解成四层：

| 层级 | 主要职责 | 关键词 |
| --- | --- | --- |
| **任务层** | 接收分析目标、定义完成条件 | 离线任务、无人值守 |
| **Agent 层** | ReAct 执行、Todo 驱动、父子协作 | Planner / Worker |
| **工具层** | 代码理解、搜索、抽取、协议访问 | Tool、MCP、Artifact |
| **调度与状态层** | 资源协调、持久化、恢复 | SQLite、状态机、可恢复执行 |

如果只记一条主线，可以记成：

> 任务目标进入系统后，由 Planner 拆成可执行子任务，Worker 调用工具产出结构化结果，状态层负责把整个执行过程变成可恢复、可追踪、可审计的离线流程。

## 一次典型任务怎么跑

下面用“分析某类接口契约”举一个简化链路：
1. 用户或任务系统给出分析目标
2. Planner 生成 Todo 和子任务边界
3. Worker 读取目标仓库、搜索相关实现、调用工具提取事实
4. 如果还需要补证据，Planner 继续派发新的 Worker
5. 结构化结果写入状态层或 Artifact
6. 所有 Todo 完成后，系统输出最终报告

这条链路里最关键的不是“模型说了什么”，而是：
- 任务有没有被外化记录
- 事实是不是通过工具拿到
- 子任务是否可恢复
- 结果能不能追溯来源

## 为什么它不是普通 Code Agent
很多现成 Code Agent 更适合：
- 交互式修改代码
- 一次性问答
- 面向单仓库的小范围辅助

Code-Miner 更偏向：
- 长时间运行
- 跨仓库分析
- 结构化知识抽取
- 强约束、强审计、强恢复能力

它不是在追求“更像人”，而是在追求“更像一套稳定的工程系统”。

---

## Code-Miner 的几个关键词

### 离线 ReAct
仍然是 Think -> Act -> Observe 的循环，但终止条件由系统规则控制，而不是只靠模型主观判断。

### Todo 外化
把“接下来还要做什么”从模型记忆里拿出来，交给系统维护。

### Planner / Worker
把全局规划和局部执行拆开，降低上下文膨胀和故障污染。

### Tool First

工具是事实源，模型是策略单元。

### Strong State

任务状态、执行进度、子任务结果和恢复机制都由系统维护，而不是依赖一次幸运的连续推理。

## 这套体系最适合什么场景

- 代码资产规模大，人工无法完整梳理
- 需要跨仓库、跨模块串联事实
- 希望长期运行，而不是临时问答
- 对稳定性、可恢复性、成本和审计有明确要求
- 需要把代码知识抽取成可复用的工程资产

## 总结
Code-Miner 的重点不在于“模型能不能更聪明地读代码”，而在于把代码分析这件事变成一条**可以长期运行、可恢复、可追踪、可控成本**的工程链路。

从这个角度看，它首先是一个工程系统，其次才是一个 Agent。
